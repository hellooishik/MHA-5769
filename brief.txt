Assessment Requirements 
Part 1: Group Project Artefact and Technical Evaluation (60%) 
Overview: 
This assessment builds on Part 1, where your group developed a conceptual AI system 
design. In Part 2, you are required to develop a working artefact and conduct a 
structured technical evaluation.  
The artefact must be implemented using Python and appropriate open-source libraries, 
reflecting the proposed system design. The focus is on demonstrating AI systems 
engineering practice, including implementation, experimentation, evaluation, and 
system-level considerations.  The accompanying report should provide a critical technical 
evaluation of the developed solution, including its performance, robustness, real-world 
applicability, and limitations.  
 
2. Assessment Task 
Your group must: 
• 
Develop a working artefact that implements a key component or subsystem 
of your proposed AI system in Part 1 of the assessment, using Python and 
open-source tools. 
• 
Implement an appropriate data processing and modelling pipeline, ensuring 
clarity, structure, and reproducibility. 
• 
Evaluate the artefact in terms of performance, robustness, and real-world 
applicability, using suitable metrics, verification and validation strategies.   
• 
Critically analyse the methodology, design decisions, and experimental 
results, identifying strengths, limitations, and trade-offs, while considering 
the broader socio-technical context, including stakeholder impact, 

 
3 
 
operational constraints, ethical implications, and real-world deployment 
conditions. 
• 
Reflect on systems engineering challenges, constraints, and opportunities 
for improvement or extension. 
Deliverable 
You must submit the following: 
• 
A single 3,000-word technical evaluation report (excluding references and 
appendices), using the Part 2 Report Submission Template available on Moodle. 
• 
Include accurate citations and a reference list in IEEE format. 
• 
There is no ±10% leeway on the stated word count. 
• 
The report must include a group contribution statement presented as a brief bullet 
list per member in the Appendix, clearly outlining individual roles and 
responsibilities. 
• 
File naming convention: use your group number only (e.g., Group_1.pdf).  
Group numbers and member details are available in the shared Excel file. 
1. Technical Evaluation Report (3000 words) 
Your report must include the following sections: 
1. Artefact Overview and Implementation 
Describe the developed artefact and its alignment with the Part 1 design, including 
a brief restatement of the problem addressed.  
2. Data and Methodology 
Describe the data used, preprocessing steps, and overall methodology. Provide 
justification for key design and implementation decisions. 
3. Experimentation and Evaluation 
Present the evaluation approach, including performance metrics, validation strategy, 
and experimental setup. Report and interpret the results. 
4. Performance, Robustness and Applicability 
Critically evaluate the artefact in terms of performance, robustness to variation or 
uncertainty, and its suitability for real-world deployment. 
5. System Engineering Considerations, Reflection and Limitations 
Discuss system-level aspects such as reproducibility, scalability, deployment, 
monitoring, and integration within a broader system context. Critically analyse the

implemented solution, including methodological limitations, technical challenges, 
and design trade-offs. Reflect on how these factors influence real-world applicability 
within a socio-technical context and propose justified improvements or extensions. 
2. Artefact (GitHub + .zip Submission) 
You must develop and manage your artefact using a version-controlled repository (e.g., 
GitHub) and submit a single compressed (.zip) file via Moodle. 
The artefact must: 
• 
Be implemented in Python using appropriate open-source libraries. 
• 
Be well-structured, readable, and appropriately commented. 
• 
Include a README file with clear instructions for execution. 
• 
Include evidence of reproducibility (e.g., environment specification such as 
requirements.txt). 
• 
Ensure that the marker(s) have been provided with access to the GitHub 
repository (e.g., via a public link). 
GitHub repository requirements: 
• 
The GitHub repository must be the primary location of your codebase. 
• 
A link to the GitHub repository must be included in the appendix of the 
report. 
• 
The repository should demonstrate ongoing development, including 
meaningful commit history and version control practices. 
• 
Use your university email to set up and manage the repository, and ensure it 
is accessible to the marker for marking purposes (e.g., via public access or 
appropriate permissions). 
• 
Commits should reflect individual contributions and development 
progression across the group. 
Zip Submission requirements:

A .zip file must be submitted via Moodle, containing the final version of the 
artefact.

The .zip file should reflect a clean, final version of the codebase, consistent 
with the GitHub repository.

The .zip file should be named using your group number (e.g., Group_1.zip).

Declaration of Use of Generative AI 
You must explicitly declare any use of generative AI tools in the preparation of this 
assessment. This includes, but is not limited to, tools such as ChatGPT, GitHub Copilot, 
Google Gemini, or other text generation, code generation, summarisation, or 
paraphrasing systems. 
• 
If generative AI tools were used, you must clearly state: 
• The name of the tool used and the purpose for which it was used (e.g., idea 
generation, language refinement, code suggestions). 
• The specific sections of the work where the tool contributed. 
This declaration should appear in a short statement at the beginning of your report. 
• 
If no generative AI tools were used, you must include a clear statement at the 
beginning of the document confirming that no generative AI tools were used in the 
preparation of the submission. 
Students remain fully responsible for the accuracy, originality, and integrity of all 
submitted work. The use of generative AI tools must comply with the university's 
academic integrity regulations, and any undeclared use may be treated as academic 
misconduct.

Assessment Success Guidance 
This assessment evaluates your ability to: 
• 
Develop a functional AI artefact using appropriate tools 
• 
Apply structured experimentation and evaluation methods 
• 
Analyse system performance, robustness, and applicability 
• 
Critically reflect on engineering decisions and limitations 
The emphasis is on engineering rigour, evaluation quality, and critical analysis, including 
systems thinking, deployment, and monitoring considerations, rather than solely 
achieving optimal model performance. 
Assessment Guidance, Support, and Formative Feedback 
Students are continually provided with opportunities to seek clarification and receive 
guidance on the assessment throughout the module. During scheduled sessions, they 
can present draft work, discuss their problems and proposed approaches, and obtain 
formative feedback to improve their final submission. In addition, Week 12 is dedicated 
to an Assessment Workshop, where students can consolidate their understanding of the 
assessment criteria, refine their work, and address any remaining questions before 
submitting their final work.

Submission Checklist  
Before you submit, ask yourself: 
• All parts of the assessment brief have been fully addressed and task 
requirements met. 
• The word count, structure, and formatting requirements have been followed. 
• Referencing is complete, accurate, and presented in IEEE style. 
• Any use of generative AI tools has been clearly and honestly declared. 
• The work has been proofread for clarity, accuracy, and consistency. 
• The report, code artefact, and all supporting files are correctly prepared and 
included. 
• The GitHub repository link is accessible to the marker and permissions are 
correctly set. 
• The artefact is executable and can be run using the provided instructions. 
• The group contribution statement is included in the Appendix. 
• All required components have been submitted via the correct platform before 
the deadline.
